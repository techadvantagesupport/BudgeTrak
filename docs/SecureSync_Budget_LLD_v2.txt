1. INTRODUCTION
================================================================================

SecureSync Budget is a personal budget management Android application built
with Jetpack Compose and Material Design 3. The application enables users to
track income and expenses, configure recurring financial entries, amortize
large purchases over time, set and manage savings goals, and visualize
spending through interactive charts. A distinctive Solari-style split-flap
display animates the user's available cash on the dashboard. The app supports
two languages (English and Spanish), multiple currency formats, encrypted
file save/load functionality, CSV import from US Bank statements, and
multi-device family synchronization via Firebase Firestore with per-field
LWW-Register CRDT conflict resolution. All domain data is persisted as JSON
files in the application's private storage directory, with user preferences
stored in SharedPreferences. The architecture follows a single-activity
Compose pattern where MainActivity serves as the central state coordinator,
delegating UI rendering to composable screen functions and data persistence
to singleton repository objects.

This Low-Level Design document describes every class, composable function,
data model, utility object, and persistence schema in the SecureSync Budget
application. Each entry includes its package, purpose, constants, fields,
methods with detailed descriptions, and relevant design notes.

Methods:

    onCreate(savedInstanceState: Bundle?)
        Entry point. Calls enableEdgeToEdge(), enters setContent block
        where all state is declared, data loaded, categories provisioned,
        budget refresh runs, and screen router dispatches to composables.

    saveIncomeSources()
        Persists incomeSources to IncomeSourceRepository.

    saveRecurringExpenses()
        Persists recurringExpenses to RecurringExpenseRepository.

    saveAmortizationEntries()
        Persists amortizationEntries to AmortizationRepository.

    saveSavingsGoals()
        Persists savingsGoals to SavingsGoalRepository.

    saveTransactions()
        Persists transactions to TransactionRepository.

    saveCategories()
        Persists categories to CategoryRepository.

    savePeriodLedger()
        Persists periodLedger to PeriodLedgerRepository.

    saveSharedSettings()
        Persists sharedSettings to SharedSettingsRepository.

    persistAvailableCash()
        Writes availableCash to SharedPreferences.

    recomputeCash()
        Deterministically recomputes availableCash from budgetStartDate,
        periodLedger, transactions, and recurringExpenses via
        BudgetCalculator.recomputeAvailableCash(). Called after sync
        merges and transaction mutations.

    recalculateBudget()
        Invokes BudgetCalculator.calculateSafeBudgetAmount() and stores
        the result. Note: safeBudgetAmount and budgetAmount are now
        derivedStateOf and auto-recalculate when inputs change.

    isBudgetAccountedExpense(txn: Transaction): Boolean
        Returns true if expense is categorized as Recurring or Amortization.

    addTransactionWithBudgetEffect(txn: Transaction)
        Adds transaction, saves, and adjusts availableCash.

    runMatchingChain(txn: Transaction)
        Sequential matching: duplicate, recurring, amortization, budget
        income. Sets linkedRecurringExpenseId or linkedAmortizationEntryId
        on match.

    Budget Period Refresh Logic:
        Calculates missed periods, records each in period ledger with
        budgetAmount, updates savings goals, persists all state.

    Supercharge Handler:
        Processes savings goal supercharge allocations with CRDT stamping.

    CRDT Stamping Pattern:
        All data mutations stamp the modified record's field _clock values
        using lamportClock.tick() and set deviceId to localDeviceId.

                .padding(start = 12.dp, bottom = 18.dp)
        )
    }

4.1  FlipSoundPlayer
--------------------------------------------------------------------------------

File:       FlipSoundPlayer.kt (131 lines)
Package:    com.syncbudget.app.sound

Description:
    Procedural audio generator and player for Solari flip card animations.
    Generates a synthetic mechanical clack at initialization and plays it
    via SoundPool with low-latency overlap support.

Constants:
Constant                Value       Purpose
---------------------------------------------------------------------------
SAMPLE_RATE             44100       Audio sample rate in Hz
DURATION_MS             45          Sound duration in milliseconds
MAX_STREAMS             6           Maximum concurrent SoundPool streams
VOLUME                  0.8f        Playback volume (left and right)

Sound Synthesis:
    1. Exponential decay envelope: exp(-t * 120)
    2. Mechanical bounce at 10ms: 0.3 * exp(-(dt^2) / 0.000002)
    3. Three sine waves (800Hz, 1200Hz, 2400Hz) + 40% white noise
    4. Combined signal scaled to 70% of Short.MAX_VALUE
    5. Encoded as 16-bit mono PCM RIFF WAV
    6. Written to cache directory, loaded into SoundPool

Methods:
    play()
        Plays the pre-generated clack sound via SoundPool. Called from
        flip card animation midpoint callbacks.

    release()
        Releases the SoundPool and deletes the cached WAV file.

6.7  SavingsSimulator (Object Singleton)
--------------------------------------------------------------------------------

File:       SavingsSimulator.kt (150 lines)
Package:    com.syncbudget.app.data

Description:
    Forward-looking cash flow simulation that determines how much savings
    buffer the user needs. Walks forward from today through all upcoming
    income deposits, expense due dates, and budget spending to find the
    point where cash balance dips lowest.

Inner Classes:
    SimResult(savingsRequired: Double, lowPointDate: LocalDate?)
    CashEvent(date: LocalDate, amount: Double, priority: Int) [private]

Methods:

    calculateSavingsRequired(incomeSources, recurringExpenses, budgetPeriod,
        budgetAmount, availableCash, resetDayOfWeek, resetDayOfMonth,
        today): SimResult

        Algorithm:
        1. Determine horizon: furthest next occurrence across all income
           sources and expenses (2-year lookahead via generateOccurrences).
           If horizon == today, return SimResult(max(0, availableCash), today).
        2. Build events list:
           a. Day-0: CashEvent(today, -availableCash, priority=1)
           b. Income occurrences: CashEvent(date, +amount, priority=0)
           c. Expense occurrences: CashEvent(date, -amount, priority=2)
           d. Period boundaries (after today): CashEvent(date, -budgetAmount,
              priority=1)
        3. Sort by (date, priority) - income before deductions before expenses
        4. Walk: balance=0, track minBalance and minDate
        5. Return SimResult(roundCents(max(0, -minBalance)), minDate or null)

    generatePeriodBoundaries(after, horizon, budgetPeriod, resetDayOfWeek,
        resetDayOfMonth): List<LocalDate> [private]

        Generates period-start dates strictly after 'after' up to 'horizon'.
        DAILY: every day from after+1
        WEEKLY: next resetDayOfWeek via TemporalAdjusters.next(), then +7
        MONTHLY: next resetDayOfMonth, then +1 month with day clamping

---------------------------------------------------------------------------

    - Budget calculations rounded to 2 decimal places

    - Decimal place limiting per currency
    - Date range validation
    - Amount sum verification for multi-category
