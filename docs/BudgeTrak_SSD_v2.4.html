1. Executive Summary

BudgeTrak is a personal budget management Android application built entirely with Jetpack Compose and Material Design 3. The application provides a comprehensive suite of financial tracking tools including income management, recurring expense tracking, amortization scheduling, savings goals with Supercharge acceleration, and full transaction management with bank import capabilities. The app features a distinctive Solari-style mechanical split-flap display for the main budget amount, complete with procedurally generated mechanical clack sound effects.

BudgeTrak supports English and Spanish localization through a CompositionLocal-based string injection system with over 800 translatable string fields organized across 16 interface groups. All primary data is persisted locally using JSON file-based storage with SafeIO atomic writes. A Family Sync feature enables multi-device synchronization (up to 5 devices) via Firebase Firestore using per-field encrypted document storage with filtered real-time snapshot listeners and Firebase Realtime Database (RTDB) for instant online/offline presence detection. Receipt photos can be captured, linked to transactions, and synced across devices via Firebase Cloud Storage. Transaction export supports ChaCha20-Poly1305 authenticated encryption with PBKDF2 key derivation for secure data portability.

  • Receipt photo capture (up to 5 per transaction) with encrypted Cloud Storage sync and 14-day pruning

2. System Architecture

2.1 High-Level Architecture

The application follows a single-Activity architecture with an AndroidViewModel holding all state and business logic, and Composable-based screen navigation:

  1. MainActivity (ComponentActivity, ~2,181 lines) serves as the sole entry point and UI-only shell. It obtains the MainViewModel via viewModel(), observes its state, and passes state and callbacks to child Composable screens. It contains no business logic, no save functions, and no mutable state of its own beyond navigation and dialog visibility.
  2. MainViewModel (AndroidViewModel, ~1,795 lines) holds all ~80 mutable state variables, all save functions, business logic, sync lifecycle management, and background loops via viewModelScope. It loads all data from JSON repositories into mutable state lists at startup and coordinates with sync, period refresh, and budget calculation systems.
  3. Navigation is string-based ("main", "settings", "transactions", "budgetConfig", "futureExpenditures", "amortization", "recurringExpenses", "family_sync", "budget_calendar", "simulation_graph", and corresponding help screens). A single mutableStateOf<String> variable controls which screen Composable is rendered.
  4. Data is in-memory: All data models are loaded into mutableStateListOf collections at startup. Data is operated on in-memory and saved back to JSON files after each mutation via SafeIO atomic writes, ensuring immediate persistence without database overhead.
  5. Budget period refresh runs every 30 seconds in the foreground and every 15 minutes in the background via BackgroundSyncWorker. It calculates missed periods since the last refresh and adds (budgetAmount x missedPeriods) to the available cash balance. Each refresh is recorded in the period ledger for deterministic cash recomputation. The refresh logic is shared via PeriodRefreshService with @Synchronized to prevent races. In sync groups, period refresh is deferred until after awaitInitialSync() completes, ensuring fresh data from Firestore before any ledger writes.
  6. System categories: Four system categories (Other, Recurring, Amortization, Recurring Income) are auto-provisioned on startup if not already present, defined by tag in DefaultCategories.kt.
  7. Theme system uses CompositionLocalProvider to inject both custom colors (SyncBudgetColors) and localized strings (AppStrings) into the Compose tree.
  8. Sync fields: All data classes carry deviceId and deleted fields for multi-device synchronization. Tombstone-based deletes preserve records with deleted=true; UI lists filter via the .active extension property. There are no per-field clock values; conflict detection uses Firestore-native field-level updates with lastEditBy tracking.
  9. Available cash is deterministically recomputed from synced data (period ledger + transactions) via BudgetCalculator.recomputeAvailableCash(), ensuring all devices converge to the same value.
  10. Auto-push: All save functions automatically push records to Firestore via SyncWriteHelper. It is impossible to create a local-only change when sync is configured.
  11. Lifecycle coordination: MainActivity.isAppActive is a static @Volatile boolean set on ON_START and cleared on ON_STOP. BackgroundSyncWorker checks this flag at the start of doWork() and returns early when the foreground is active, avoiding duplicate period refresh and widget updates that the foreground already handles.

3. Solari Flip Display System

3.1 Overview

The centerpiece of the BudgeTrak dashboard is a Solari-style mechanical split-flap display that animates the current budget amount. This system comprises three Composable components and a procedural audio engine that work together to produce a faithful digital recreation of the iconic departure-board aesthetic.

3.2 FlipDisplay Composable (FlipDisplay.kt — 257 lines)

FlipDisplay is the top-level compositor that arranges individual flip cards into a complete budget readout. It handles:

  • Currency-aware formatting with correct decimal places per currency
  • Responsive card sizing using BoxWithConstraints to fit available width
  • Leading zero suppression (blank cards for leading zeros)
  • Sign/currency combination card that displays currency symbol alone for positive values and "-" + symbol for negative values
  • Decimal point rendering as bare text between whole and decimal cards
  • Optional bottom label text
  • Proportional font scaling based on computed card dimensions

Display layout: [Sign+Currency] [Digit] ... [Digit] [.] [Digit] [Digit]

3.3 FlipChar Composable (FlipChar.kt — 293 lines)

FlipChar renders an animated flip card for arbitrary string values. It is used for the combined sign + currency card. The flip animation consists of:

  • A 250ms flip duration (FLIP_DURATION_MS)
  • 3D rotation around the horizontal center axis using graphicsLayer
  • Four visual layers: static top half, static bottom half, flipping top-away panel, and flipping bottom-reveal panel
  • Gradient overlays for depth simulation
  • LinearOutSlowInEasing interpolation for natural deceleration
  • Sound callback at animation midpoint

Multi-character currency symbols (e.g., "Fr") are split across two lines within the card: sign on top, symbol on bottom.

3.4 FlipDigit Composable (FlipDigit.kt — 316 lines)

FlipDigit renders an animated flip card for single digits 0-9 with support for a blank state (digit value -1). The animation mechanics are identical to FlipChar but optimized for the digit use case:

  • Integer-based target values instead of string-based
  • Blank card rendering when targetDigit is -1
  • Same 250ms flip with 3D rotation and gradient overlays
  • Monospace FlipFontFamily for consistent digit widths

3.5 FlipSoundPlayer (FlipSoundPlayer.kt — 134 lines)

FlipSoundPlayer generates and plays a synthetic mechanical clack sound that accompanies each flip card animation. The sound is procedurally generated at initialization rather than loaded from a bundled asset.

Audio Parameters

Bit Depth16-bit signed PCM
ChannelsMono

Sound Synthesis Algorithm

  1. Fast-attack exponential decay envelope: exp(-t * 120)
  2. Secondary mechanical bounce at t=10ms: 0.3 * exp(-(dt^2) / 0.000002)
  3. Band-limited noise combining three sine waves at 800Hz, 1200Hz, and 2400Hz with random phase offsets, plus 40% white noise component
  4. Combined envelope applied to signal, scaled to 70% of Short.MAX_VALUE

The generated audio is encoded as a RIFF WAV file, written to the app cache directory, and loaded into a SoundPool for low-latency playback.

4.2 BudgetCalculator (BudgetCalculator.kt — 482 lines)

BudgetCalculator is a singleton object that normalizes all income sources and recurring expenses to the configured budget period using an occurrence-based projection system.

Primary Functions

FunctionPurpose
generateOccurrences()Generates all occurrence dates for a repeat schedule within a given date range. Supports all 6 RepeatType values with efficient fast-forward for start dates before range.
calculateSafeBudgetAmount()Projects total income and total expenses over a 1-year window, counts periods in that window, and computes per-period discretionary budget.
countPeriodsCompleted()Counts budget periods between two dates using ChronoUnit (DAYS, WEEKS, MONTHS).

Safe Budget Formula

periodsPerYear = DAILY:365.25, WEEKLY:365.25/7, MONTHLY:12.0
totalAnnualIncome = SUM(src.amount x theoreticalAnnualOccurrences(src.repeatType, src.repeatInterval))
totalAnnualExpenses = SUM(re.amount x theoreticalAnnualOccurrences(re.repeatType, re.repeatInterval))
safeBudgetAmount = max(0, (totalAnnualIncome - totalAnnualExpenses) / periodsPerYear)

Effective Budget Amount

base = isManualBudgetEnabled ? manualBudgetAmount : safeBudgetAmount
budgetAmount = max(0, base - amortDeductions - savingsDeductions - accelDeductions)

Amortization Deductions

elapsed = countPeriodsCompleted(startDate, today, budgetPeriod)
deduction = roundCents(amount * (elapsed+1) / totalPeriods) - roundCents(amount * elapsed / totalPeriods)

Savings Goal Deductions

  • Target-date: remaining / periodsUntilTarget
  • Fixed-contribution: min(contributionPerPeriod, remaining)

4.3 Budget Period Refresh

Period refresh logic is encapsulated in PeriodRefreshService (275 lines), shared by both the foreground LaunchedEffect (every 30 seconds) and the background BackgroundSyncWorker (every 15 minutes). A @Synchronized annotation prevents foreground/background race conditions.

timezone = if (syncConfigured && familyTimezone.isNotEmpty()) ZoneId.of(familyTimezone) else null
currentPeriod = BudgetCalculator.currentPeriodStart(budgetPeriod, resetDayOfWeek, resetDayOfMonth, timezone, resetHour)
missedPeriods = countPeriodsCompleted(lastRefreshDate, currentPeriod, budgetPeriod)

When missedPeriods > 0:

  1. Compute budgetAmount internally via BudgetCalculator.computeFullBudgetAmount()
  2. Create one ledger entry per missed period with computed budgetAmount as appliedAmount
  3. Update each non-paused savings goal's totalSavedSoFar per-period
  4. Update RE setAsideSoFar per-period (reset if occurrence reached, else increment)
  5. Update lastRefreshDate to today
  6. Call recomputeCash()
  7. Save updated state to SharedPreferences and repositories

Foreground vs Background Execution

  • Foreground (LaunchedEffect): Applies results directly to Compose SnapshotStateLists in MainViewModel
  • Background (BackgroundSyncWorker): Pushes ledger entries via FirestoreDocService.createDocIfAbsent() and SG/RE updates via FirestoreDocService.updateFields(), then saves to disk via repositories. Publishes availableCash in device metadata via fingerprintJson for cross-device cash comparison even when the app is in the background.
  • Foreground guard: BackgroundSyncWorker checks MainActivity.isAppActive at the start of doWork() and returns early when the foreground is active, avoiding duplicate refresh and widget updates.
Timezone rule: When in a sync group, ALL period calculations use sharedSettings.familyTimezone to ensure all devices see the same period boundaries regardless of local timezone.
Sync deferral: Period refresh is deferred until awaitInitialSync() completes when sync is configured. This suspending call waits for all 8 collection listeners to deliver their initial snapshots (completes instantly for filtered sessions with no new data, up to 30s for large datasets). This prevents offline devices from overwriting correct period ledger entries with stale budget data. Solo users skip this wait.

4.4 Manual Budget Override

Users may optionally bypass the calculated safe budget with a manual override:

  • isManualBudgetEnabled (Boolean) — toggles override mode
  • manualBudgetAmount (Float) — user-specified per-period amount

When enabled, the manual amount replaces the calculated safe budget for all period refresh calculations. If the manual amount exceeds the safe budget amount, the savings required calculation on the Recurring Expenses page is disabled with a warning message.

4.5 Deterministic Cash Recomputation

Available cash is deterministically recomputed from synced data via BudgetCalculator.recomputeAvailableCash():

Step 1: Sum Period Ledger Credits

cash = SUM(entry.appliedAmount) for entries where date >= budgetStartDate
       deduped by date (keep one entry per epoch day)

Step 2: Apply Transaction Effects

EXPENSE branches (in order of precedence):

  1. linkedSavingsGoalId != null OR linkedSavingsGoalAmount > 0 → SKIP (money from savings)
  2. linkedAmortizationEntryId != null → SKIP (budget-accounted via AE deduction)
  3. amortizationAppliedAmount > 0 → cash -= max(0, amount - amortizationAppliedAmount) (deleted AE remainder)
  4. linkedRecurringExpenseId != null with linkedRecurringExpenseAmount > 0 → cash += (rememberedAmount - amount)
  5. linkedRecurringExpenseAmount > 0 (but no ID = deleted RE) → cash += (rememberedAmount - amount)
  6. Regular expense → cash -= amount

INCOME branches:

  1. linkedIncomeSourceId != null: FIXED = no effect; ACTUAL = cash += (amount - rememberedAmount); ACTUAL_ADJUST = no effect
  2. linkedIncomeSourceAmount > 0 (deleted source): ACTUAL mode: cash += (amount - rememberedAmount); FIXED/ADJUST: no effect
  3. isBudgetIncome == true → no effect (already budgeted)
  4. Non-budget income → cash += amount

Step 3: Return roundCents(cash)

Key Invariant: All devices with identical synced data compute identical availableCash. The formula is deterministic given the same inputs.

5.3 Supercharge Dialog

The Supercharge feature allows users to allocate surplus cash from the current budget period to accelerate their savings goals. Workflow:

  1. Display remaining cash available for the current period
  2. For each active (unpaused, not-yet-complete) savings goal: show toggle and mode selector
    • REDUCE_CONTRIBUTIONS: Apply extra payment and recalculate lower per-period contributions
    • ACHIEVE_SOONER: Apply extra payment and show new earlier completion date
  3. Preview section shows per-goal impact before confirming
  4. On confirm: update totalSavedSoFar, deduct from availableCash, recalculate budget, save all changes

5.4 Persistence

Savings goals are stored in future_expenditures.json (legacy filename). The repository supports legacy field migration: "description" → "name", "amount" → "targetAmount".

6.2 Amortization Logic

Amortization allows users to spread the budget impact of a large purchase across multiple budget periods. The per-period deduction is: deduction = amount / totalPeriods

An entry is active when: elapsed < totalPeriods AND NOT isPaused

Progress display shows "X of Y [periods] complete" with a visual progress bar and the per-period deduction amount.

6.3 Auto-Matching on Import

When transactions are imported, the DuplicateDetector matches against existing amortization entries using amount matching (percentage OR dollar tolerance) and merchant name matching (common substring). If matched, the user is prompted: "Yes, Amortization" or "No, Regular Expense". Linked transactions store the linkedAmortizationEntryId.

7.4 Savings Required Cash Flow Simulation

SavingsSimulator.kt (517 lines) performs a forward-looking cash flow simulation to determine the savings buffer a user needs.

Algorithm

  1. Determine horizon: the furthest next occurrence across all income sources and recurring expenses (up to 2 years)
  2. Generate all events: income occurrences (+amount, priority 0), budget period deductions (-budgetAmount, priority 1), expense occurrences (-amount, priority 2), day-0 event (-availableCash, priority 1)
  3. Sort events by (date, priority) so income arrives before deductions before expenses on the same day
  4. Walk the timeline: start with balance = 0, apply each event, track minimum balance and date
  5. savingsRequired = max(0, -minimumBalance)

Returns SimResult(savingsRequired, lowPointDate).

8.4 Linked Transaction Lifecycle

KEY DESIGN RULE: Delete preserves remembered amounts (expense already paid). Manual unlink clears them (linked-in-error). This asymmetry is intentional and applies to ALL link types (RE, IS, AE, SG).

Link to Recurring Expense

  • Link: Set linkedRecurringExpenseId=re.id, linkedRecurringExpenseAmount=re.amount
  • Cash effect: cash += (rememberedAmount - txn.amount)
  • RE deleted: ID=null, amount PRESERVED (expense already paid)
  • Manual unlink: ID=null, amount=0.0 (linked-in-error, full amount applies)

Link to Income Source

  • Link: Set linkedIncomeSourceId=src.id, linkedIncomeSourceAmount=src.amount
  • Cash effect: FIXED=none, ACTUAL=cash += (txn.amount - rememberedAmount), ADJUST=none
  • IS deleted: ID=null, amount PRESERVED
  • Manual unlink: ID=null, amount=0.0

Link to Amortization Entry

  • Link: Set linkedAmortizationEntryId=ae.id
  • While linked: Transaction excluded from cash calc entirely
  • AE deleted: ID=null, amortizationAppliedAmount=cumulative deduction at delete
  • Manual unlink: ID=null, amortizationAppliedAmount=0.0
  • Cash effect (deleted AE): cash -= max(0, amount - amortizationAppliedAmount)

Link to Savings Goal

  • Link: Set ID + amount, goal.totalSavedSoFar += amount
  • While linked: Transaction excluded from cash (money in savings)
  • SG deleted: ID=null, amount PRESERVED (money already spent from savings)
  • Manual unlink: ID=null, amount=0.0, goal.totalSavedSoFar += amount (restore funds)

16. Family Sync System (Firestore-Native)

16.1 Overview

Family Sync enables multi-device budget synchronization for households of up to 5 devices. The system uses Firebase Firestore as the storage and transport layer with per-field encrypted document storage, filtered real-time snapshot listeners with per-collection updatedAt cursors, and Firebase Realtime Database (RTDB) for instant online/offline presence detection. This replaces the earlier CRDT-based sync system (removed in v2.1) with a simpler, more reliable architecture.

Architecture change (v2.1 → v2.2): The hand-rolled CRDT sync system (~4,000 lines: LamportClock, DeltaBuilder, DeltaSerializer, CrdtMerge, SnapshotManager, SyncEngine, IntegrityChecker, PeriodLedgerCorrector, SyncFileLock) was replaced with Firestore-native document storage (~1,500 lines: EncryptedDocSerializer, FirestoreDocService, FirestoreDocSync, SyncWriteHelper). Data classes no longer carry per-field _clock values.
Architecture change (v2.3 → v2.4): Major sync cost optimization. Listeners now use whereGreaterThan("updatedAt", cursor) with per-collection cursors, reducing Firestore read costs from ~$50K/mo to ~$160/mo at 100K devices. RTDB presence replaces Firestore device polling for online/offline detection. Health check polling loop eliminated in favor of one-time startup checks. Image ledger collection listener replaces 60s polling. Immediate photo download in onBatchChanged. awaitInitialSync() completion signal replaces fixed delays.

16.2 Per-Field Encryption

Each business field is encrypted separately as enc_fieldName in Firestore documents. Metadata fields (deviceId, updatedAt, deleted, lastEditBy) remain in plaintext. Uses CryptoHelper.encryptWithKey (ChaCha20-Poly1305 with direct 256-bit key).

This design ensures:

  • Field-level access control: metadata is readable without the key
  • Field-level updates: only changed encrypted fields need to be transmitted
  • End-to-end encryption: Firestore administrators cannot read business data
  • Small update payloads: editing one field sends only that enc_* field

16.4 Real-Time Sync via Filtered Listeners

8 persistent snapshot listeners (one per collection) provide real-time synchronization. Each listener uses a per-collection updatedAt cursor to filter documents, ensuring only changed records are read on reconnection:

  • Managed via DisposableEffect, survive app pause/resume
  • Each listener queries whereGreaterThan("updatedAt", cursor) where cursor is the last-seen updatedAt timestamp for that collection, stored in SharedPreferences
  • Cursor saved after onBatchChanged completes processing
  • Falls back to full read (cursor = epoch 0) on reinstall or first sync
  • After >30 minutes of app inactivity, listeners reattach with saved cursors, reading only documents changed during the gap
  • Batched listener callbacks: save once per collection, not per record
  • Persistent listeners mean no re-delivery on warm resume

awaitInitialSync() Completion Signal

FirestoreDocSync tracks which collections have delivered their initial snapshot via markCollectionDelivered(). The awaitInitialSync(30_000) suspending function waits until all 8 collections have reported. This replaces fixed 3s/5s delays with a precise signal:

  • Completes instantly on filtered sessions when no new data arrives (cursors are current)
  • Waits up to 30 seconds for large initial datasets (e.g., first sync after joining a group)
  • Defers period refresh, migrations, and save operations until data is populated

Cost Impact

Per-collection cursor filtering reduces Firestore document reads dramatically. A device that reconnects after 30 minutes of inactivity reads only documents changed during the gap (typically 0-5) instead of all documents in all 8 collections (potentially thousands). At scale: estimated ~$50K/month at 100K devices with unfiltered listeners reduced to ~$160/month with cursor-based filtering.

Sync Data Processing (SyncMergeProcessor)

Merge logic for incoming sync events is extracted into SyncMergeProcessor (263 lines), shared by both the foreground listener callbacks and the background BackgroundSyncWorker.

  • Category dedup: tag-based deduplication for system categories
  • Conflict detection: checks lastEditBy vs localPendingEdits
  • Settings application: applies shared settings (budget period, currency, etc.)
  • Standard merge: add-or-replace for all other record types

Returns a MergeResult containing merged data plus side effects (categories to delete, conflicts to push back, settings to apply). The foreground applies results directly to Compose SnapshotStateLists; the background saves results to disk via repositories.

Listener Lifecycle

  1. On sync setup (group join/create): attach 8 filtered listeners via FirestoreDocSync
  2. Each listener callback: decrypt incoming docs (enc hash skip for unchanged), diff against lastKnownState, merge via SyncMergeProcessor, apply changes to in-memory state
  3. Save updated state to local JSON repositories
  4. Save per-collection cursor to SharedPreferences
  5. Recompute available cash
  6. On dispose (group leave/dissolve): detach all listeners

Immediate Photo Download

onBatchChanged checks newly-arrived transaction documents for receipt IDs that reference files not present locally. Missing receipt files are downloaded from Cloud Storage immediately within the listener callback, rather than waiting for a periodic sync cycle.

16.5 Conflict Detection

The system detects concurrent edits using two mechanisms:

  • lastEditBy (plaintext field): tracks which device last edited each document
  • localPendingEdits (persisted to SharedPreferences): tracks unpushed offline edits on this device

Conflict scenario:

  1. Device A edits a record, stores in localPendingEdits
  2. Before push completes, listener receives change with lastEditBy != us
  3. Since we have a pending edit for the same doc → conflict detected
  4. Conflicted transactions flagged isUserCategorized = false (unverified in existing UI)

Different-field concurrent edits are preserved automatically because each device uses update() with only the specific enc_* fields it changed, not the entire document.

16.6 Integrity Check

Integrity verification runs once on startup (not periodically). All health checks (group dissolution, device removal, subscription expiry, integrity) are consolidated into a single group document read via getGroupHealthStatus():

  1. Read local record IDs from Firestore local cache via Source.CACHE (zero network reads)
  2. Compare local record IDs against cached Firestore state
  3. Push any local-only records that are missing from Firestore
  4. Run recomputeCash() to ensure deterministic cash convergence
No health check loop: The v2.3 5-minute polling loop has been eliminated. All checks run once on startup with a single group document read. RTDB presence (Section 16.14) handles online/offline detection. Integrity is ensured by the filtered listeners which deliver all changes since the last cursor.
No peer fingerprints: The fingerprint-based cash comparison between peers has been removed. It was log-only diagnostic that consumed Firestore reads for no repair action. Firestore is the single source of truth, and cursor-based listeners ensure local state converges.

16.7 Period Ledger Sync

Period ledger entries require special handling to prevent overwrites:

  • Create-if-absent via Firestore transaction: the first device to create a ledger entry for a given period wins
  • Deferred refresh: period refresh waits for awaitInitialSync() to complete before executing, preventing offline devices from writing stale entries
  • Solo skip: devices not in a sync group skip the wait entirely
  • Dedup: entries are deduplicated by epoch day (one entry per period start date)

16.13 Deterministic Cash

Available cash is not synced directly. Instead, it is deterministically recomputed from synced data:

availableCash = sum(periodLedgerCredits) + sum(transactionEffects)

This ensures all devices with the same Firestore-synced data arrive at the same cash value, eliminating sync drift. The integrity check (Section 16.6) runs recomputeCash() on startup to verify convergence.

16.14 RTDB Presence System

RealtimePresenceService.kt (180 lines) uses Firebase Realtime Database for instant online/offline detection, replacing Firestore device polling entirely.

How It Works

  1. On app launch, write online: true to RTDB node syncDevices/{groupId}/{deviceId}
  2. Register server-side onDisconnect() handler that writes online: false + timestamp when the device disconnects (even on crash, battery death, or network loss)
  3. All devices listen to the syncDevices/{groupId} node for real-time online/offline status of all group members

DeviceInfo Fields (from RTDB)

FieldTypeDescription

Benefits Over Firestore Polling

  • Instant detection: Online/offline transitions visible within seconds (vs. 60-second Firestore metadata polling)
  • Server-side reliability: onDisconnect() fires even on unclean shutdown
  • Zero Firestore reads: Device presence no longer consumes Firestore read quota
  • Device capabilities: photoCapable and uploadSpeedBps written to RTDB, eliminating FirestoreService.getDevices() calls from receipt sync path. ReceiptSyncManager takes DeviceInfo instead of DeviceRecord.

Sync Status Display

  • UI label changed from "Last synced" to "Last data sent/received"
  • Sync time display uses snapshotFlow { lastSyncActivity } for immediate update on data change, plus 10-second ticker for elapsed time refresh
  • Group member status indicators: green dot (<2min since last seen), yellow (<10min), gray (older)
RTDB prerequisite: Firebase Realtime Database must be enabled in the Firebase Console and google-services.json must be re-downloaded to include the RTDB URL. The firebase-database dependency is required in build.gradle.kts.

  2. Remove Firestore ledger entry
  3. Delete Cloud Storage file
  4. Delete local file and thumbnail
  5. Remove from pending upload queue

17.7 Orphan Cloud Storage Scan

Safety net for leftover cloud files not referenced by any ledger entry. Runs on admin startup, time-gated to once per 30 days (lastAdminCleanup in sync prefs), with a 10-minute time guard to avoid deleting files from in-progress uploads by other devices.

  • Download retry: 3 real failures (network errors don't count) → replace ledger entry with request

23. Error Handling

23.1 File I/O Error Handling

  • All repository load operations check file existence before reading
  • Return emptyList() if file does not exist or is blank
  • SafeIO atomic writes (temp + rename) prevent data corruption on interrupted writes
  • Sync fields loaded with backward-compatible defaults

23.2 Import Error Handling

  • CsvParser returns CsvParseResult with nullable error field
  • Line-level parse errors include line number and exception message
  • Header validation for native CSV format
  • Empty file detection
  • Partial results preserved (transactions parsed before error)
  • Generic CSV auto-detection with fallback scoring

23.3 Encryption Error Handling

  • Minimum data size validation (SALT_LENGTH + NONCE_LENGTH + 1)
  • Poly1305 authentication tag verification (throws on tampered data)
  • Invalid password results in authentication failure exception

23.4 Data Validation

  • Transaction IDs generated with collision avoidance (retry loop)
  • Amount tolerance checks use both percentage and absolute thresholds
  • Date boundary coercion (day of month clamped to month length)
  • Category deletion requires reassignment if transactions reference it
  • Savings goal contributions capped at remaining amount
  • Amortization periods clamped to [0, totalPeriods]
  • Budget period start dates handle month-length variations
  • Budget calculations rounded to 2 decimal places

23.5 Sync Error Handling

  • Persistent filtered listeners reconnect automatically on error; cursors ensure no data loss on reattach
  • Enc hash skip prevents redundant decryption of unchanged documents
  • Firestore operations wrapped in try-catch
  • Conflict detection via lastEditBy + localPendingEdits
  • Period ledger create-if-absent via Firestore transaction (first writer wins)
  • Receipt upload queue persisted to disk for crash safety
  • Orphan Cloud Storage scan with 10-minute time guard
  • RTDB onDisconnect() ensures offline status even on unclean shutdown
  • awaitInitialSync() timeout (30s) prevents indefinite blocking
