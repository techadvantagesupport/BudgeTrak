1. Introduction

BudgeTrak is a personal budget management Android application built with Jetpack Compose and Material Design 3. The application enables users to track income and expenses, configure recurring financial entries, amortize large purchases over time, set and manage savings goals, simulate cash-flow projections, capture receipt photos, and visualize spending through interactive charts. A distinctive Solari-style split-flap display animates the user's available cash on the dashboard.

The app supports two languages (English and Spanish), multiple currency formats, transaction export to CSV / Excel (.xlsx) / PDF expense report, CSV import from any bank with auto-detect or specific US Bank format, automatic encrypted backups in Settings, a home screen widget with quick-add transactions, and multi-device synchronization via SYNC, the Firestore-native per-document encrypted sync feature. All domain data is persisted as JSON files in the application's private storage directory, with user preferences stored in SharedPreferences.

The architecture follows a single-activity Compose pattern with an AndroidViewModel (MainViewModel) that holds all application state and business logic. MainActivity is a thin UI-only shell (~2,327 lines) that delegates to composable screen functions and includes a LoadingScreen composable gating on dataLoaded, while MainViewModel (~2,171 lines) manages ~80 state variables, save functions, sync lifecycle, background loops, and all business logic via viewModelScope. Data loading runs asynchronously on Dispatchers.IO with a learned-timing progress bar.

v2.2 Architecture Change: Firestore-Native Sync

In v2.2, the hand-rolled CRDT sync system (~4,000 lines: LamportClock, DeltaBuilder, DeltaSerializer, CrdtMerge, SnapshotManager, SyncEngine, IntegrityChecker, PeriodLedgerCorrector, SyncFileLock) was replaced with Firestore-native document storage (~1,860 lines). Each record is a separate Firestore document with per-field encryption. Firestore handles transport, offline queue, retry, and ordering. Data classes no longer carry per-field _clock values; only deviceId and deleted remain as sync fields. Real-time snapshot listeners replace the old periodic pull/push cycle. Receipt photo sync via Cloud Storage was added.

v2.3 Architecture Change: Shared Services & Background Period Refresh

In v2.3, sync merge logic and period refresh logic were extracted from MainActivity into shared services (SyncMergeProcessor, PeriodRefreshService) usable by both the foreground activity and a new BackgroundSyncWorker. The WidgetRefreshWorker was removed and replaced by BackgroundSyncWorker, which performs a short-lived Firestore sync, period refresh, cash recompute, and Firestore push — all from the background. Net result: ~220 lines moved out of MainActivity into 3 new files totaling 984 lines, and 1 file (105 lines) removed.

v2.4 Architecture Change: ViewModel Extraction, RTDB Presence, Filtered Listeners

In v2.4, all application state and business logic were extracted from MainActivity.setContent into MainViewModel (AndroidViewModel). MainActivity is now UI-only (~2,181 lines); MainViewModel holds all ~80 state vars, save functions, business logic, sync lifecycle, and background loops via viewModelScope. Sync infrastructure was overhauled: RTDB Presence replaces Firestore device polling with instant online/offline detection via RealtimePresenceService. Filtered listeners with per-collection updatedAt cursors drastically reduce Firestore read costs. awaitInitialSync() replaces fixed delays with a CompletableDeferred that completes when all 8 collection listeners have delivered. The health check loop is eliminated — all checks run once on startup after awaitInitialSync(). Integrity checks now use Source.CACHE for zero-network local Firestore cache reads. A persistent image ledger listener replaces receipt sync polling. Background receipt sync runs in BackgroundSyncWorker. Tombstone cleanup is time-gated to 30 days, and 14-day receipt pruning is simplified with no CAS. The firebase-database and lifecycle-viewmodel-compose:2.8.6 dependencies were added.

v2.5 Architecture Change: Async Data Loading, Loading Screen, Maintenance Consolidation, Transaction Archiving

In v2.5, MainViewModel.init was restructured for async data loading: seven repository loads run sequentially on Dispatchers.IO with per-segment progress tracking, and all addAll() calls plus dataLoaded = true execute on Dispatchers.Main. A LoadingScreen composable (app icon + title + LinearProgressIndicator) gates the UI via if (!vm.dataLoaded). A learned-timing progress bar stores segment times in SharedPreferences with EMA smoothing and a 60fps ticker interpolating within segments. Back = Home: BackHandler { moveTaskToBack(true) } when on the dashboard. Synchronous recomputeCash(): removed the viewModelScope.launch(Dispatchers.Default) wrapper; now computes inline. MainViewModel companion object exposes a @Volatile WeakReference<MainViewModel> for BackgroundSyncWorker. Consolidated runPeriodicMaintenance(): a single 24h-gated suspend function called from onResume containing backup check, integrity check, receipt orphan cleanup, receipt pruning, and admin tombstone cleanup. Extracted runIntegrityCheck(): reusable private suspend function comparing local IDs against Firestore cache. Three-tier BackgroundSyncWorker: checks isAppActive (skip), then MainViewModel.instance (restart dead listeners only), then full sync when ViewModel is null. Calculated period refresh: sleeps until nextBoundary + 60s instead of fixed 30s delay. Lifecycle observer moved below loading gate. Transaction archiving: archiveThreshold setting, archiveCutoffDate on SharedSettings, applyArchiveCutoff() moves old transactions to a separate archive file.

v2.5 Maintenance Updates (April 2026)

The v2.5 line received several reliability and observability updates that did not warrant a version bump:

  • SYNC rebrand: the multi-device feature was renamed from "Family Sync" to "SYNC" (always all-caps as a brand mark). The Settings button label, the page header, and all in-app strings were updated. Code symbols (FamilySyncScreen → SyncScreen, FamilySyncHelpScreen → SyncHelpScreen, the familySync/familySyncDescription string fields, the familySyncHelp field, and navigation routes "family_sync"/"family_sync_help" → "sync"/"sync_help") were renamed to match. The "Future Expenditures" terminology was likewise swept to "Savings Goals" everywhere except the local persistence file (future_expenditures.json) and the corresponding SharedSettings/Firestore field (familyTimezone) which are preserved for backwards compatibility.
  • Two-layer consistency check: runConsistencyCheck() in MainViewModel performs Layer 1 (per-collection countActiveDocs() aggregation vs. local .active.size; cursor cleared on mismatch) and Layer 2 (cashHash majority vote, with checkpoints written to deviceChecksums on the group doc and a 1-hour confirmation gate before any re-read recovery is triggered). Cash hash uses a hex digest (availableCash.toString().hashCode().toString(16)), not the raw value, to keep financial data out of Crashlytics and the cloud-stored checksum doc. Mismatch recheck runs from both onResume and BackgroundSyncWorker Tier 3 step 4b.
  • Echo suppression: cursor advances even for echo-filtered batches (both pure-echo and mixed). Background pushes persist their pushed doc keys to sync_engine prefs (bgPushKeys) so the next foreground FirestoreDocSync can filter them via recentPushes + lastEditBy. Eliminates wasted Firestore reads + decryption cycles. The encryption-hash skip cache (enc_hash_cache.json) is now populated at push time as well as at read time, so own writes do not trigger re-decryption when listeners later deliver the echo.
  • App Check token reliability: proactive token refresh now runs from four places — onResume, onAvailable (network restored), a 45-minute ViewModel keep-alive coroutine, and all three tiers of BackgroundSyncWorker (Tier 2 when the ViewModel is alive, Tier 3 when the process is restarted). The threshold is "within 35 min of expiry" (greater than the max Doze worker interval). All getAppCheckToken() calls are wrapped with withTimeoutOrNull(10–15s) to prevent hangs when Play Integrity itself is unavailable. triggerFullRestart() is the recovery path on PERMISSION_DENIED: it stops every listener, force-refreshes the App Check token (with timeout), waits for the refresh, and starts all listeners fresh on the SDK's new auth state. The 30-second debounce prevents recovery storms.
  • Crashlytics observability: BudgeTrakApplication.syncEvent(msg) logs key sync lifecycle events (listener start/stop, recovery, worker tiers) to the Crashlytics breadcrumb log + logcat. BudgeTrakApplication.updateDiagKeys(map) sets custom keys (cashDigest, listenerStatus, activeDevices, txnCount, etc.) attached to every future crash and non-fatal. A daily HEALTH_BEACON non-fatal is recorded from runPeriodicMaintenance() so production devices remain visible in the Crashlytics dashboard even when nothing is wrong. BigQuery streaming export is active.
  • Real-time SYNC eviction: device removal, leave, and dissolution events propagate via Firestore in real time, surfacing as a one-shot dashboard popup (syncEvictionMessage) the next time the user opens the app.
  • Admin claim voting: when the admin device is unavailable, any subscriber can claim the admin role; other members have 24 hours to object before the claim is approved. Implemented as a single-document workflow on the group doc with real-time listener delivery.
  • TTL fix: group documents now use a separate expiresAt Firestore Timestamp field (refreshed to now + 90 days on every launch) for the TTL policy, replacing the earlier broken use of lastActivity (a server-set timestamp that was always in the past, causing immediate TTL eligibility).
  • Subscription expiry dashboard popup: replaces the earlier notification-based approach. Shown once per session when the admin's subscription has expired, with the 7-day grace period and admin-claim instructions inline.
  • Match dialog improvements: ranked multi-candidate match dialogs (the closest entry pre-selected at the top, ranked by date and amount distance). Recurring expense and income source dialogs now show the nearest expected occurrence date alongside the new transaction's date.
  • Auto-Capitalize: a Settings toggle (default on) that formats merchant names and descriptions in Title Case as the user finishes each word.
  • Match-character normalization: merchant matching strips non-alphanumeric characters before comparing, so "Wal-Mart" and "Walmart" match.
  • Solo user improvements: Tier 2 and Tier 3 of BackgroundSyncWorker, Firebase Auth gating, App Check refresh, and a tombstone purge in runPeriodicMaintenance() are now all gated on isSyncConfigured/groupId != null. Solo users skip Auth, App Check, RTDB, and Firestore entirely; only period refresh and widget update run.

Learned-Timing Progress Bar (v2.5)

Stored segment times in SharedPreferences (loadSegTime_0..6). Default times: [50, 10, 5, 5, 5, 5, 5] ms. EMA update formula: (4 * old + new) / 5; first run saves actual time directly. A 60fps ticker coroutine interpolates loadProgress within segments using a boundaries[] array proportional to stored times. Progress advances smoothly from 0f to 1f as each repository load completes.
saveTransactions(hint?) / saveCategories(hint?) / saveIncomeSources(hint?) / etc.
Persists data to the corresponding repository. Accept optional hint: List<T>? = null. Hint passed: push exactly those records via SyncWriteHelper. No hint: snapshot-diff safety net in FirestoreDocSync finds changes automatically. Each save function calls the repository's save() then pushes via SyncWriteHelper. Updates lastSyncActivity when records are pushed.
recomputeCash()

configureSyncGroup()

onBatchChanged(events: List<DataChangeEvent>)
Receives incoming changes from real-time Firestore listeners. Delegates to SyncMergeProcessor.processBatch() which returns a MergeResult. Applies non-null collection lists from MergeResult to Compose state. Handles side effects: pushes conflicted transactions back to Firestore, deletes remapped categories from Firestore. Deferred Period Refresh
Period refresh is deferred until after awaitInitialSync() completes (suspend, not blocking) to allow incoming period ledger entries to arrive. Solo users (not in sync group) skip the wait. Calls PeriodRefreshService.refreshIfNeeded() which handles ledger creation, SG accrual, RE set-aside, cash recompute, and SharedPrefs update. Updates in-memory state from RefreshResult, pushes changed records (ledger via createDocIfAbsent(), SG and RE via updateFields()) via SyncWriteHelper, and updates lastSyncActivity.
startSyncSetup()
Launched via viewModelScope.launch after configureSyncGroup(). Gates on snapshotFlow { dataLoaded }.first { it } (v2.5). Performs one-time startup tasks:
  1. Fetches initial device list from Firestore.
  2. Writes device metadata once (photoCapable, appSyncVersion, minSyncVersion). Does NOT write lastSeen — RTDB presence handles that.
  3. Registers FCM token.
  4. Runs one-time migrations (skeleton categories, RE setaside backfill, native docs push, per-field encryption push).
  5. Attaches persistent image ledger listener for receipt downloads, possession marking, and recovery re-uploads.
  6. Waits for awaitInitialSync() (up to 30s), then runs runIntegrityCheck() + recomputeCash().
v2.5: Admin tombstone/orphan cleanup, receipt orphan cleanup, and one-time startup listener restart check were removed from startSyncSetup() and consolidated into runPeriodicMaintenance(). Simulation trace and debug dump were removed from init entirely (dump is now button-only).
Startup Health Check (One-Time)
Runs once after awaitInitialSync() completes. FirestoreService.getGroupHealthStatus() returns GroupHealthStatus(isDissolved, subscriptionExpiry) from a single group doc read. isDeviceRemoved() is a separate single doc read. No loop. Calls runIntegrityCheck() then recomputeCash().
runIntegrityCheck() (v2.5)
Private suspend function extracted in v2.5. Compares local record IDs against the Firestore local cache via FirestoreDocService.readDocIdsFromCache() (Source.CACHE, zero network cost). Pushes any local-only records missing from cache via the corresponding SyncWriteHelper.push*() methods. Called from both the startup sync flow (after awaitInitialSync()) and runPeriodicMaintenance().
runPeriodicMaintenance() (v2.5)
Private suspend function called from onResume when lastMaintenanceCheck is older than 24 hours. Consolidates previously scattered maintenance tasks into a single entry point:
  1. Backup check: Prompts for backup if overdue.
  2. Integrity check + recomputeCash: Calls runIntegrityCheck() then recomputeCash().
  3. Receipt orphan cleanup: Removes orphaned receipt references (solo users) and orphaned local files.
  4. Receipt storage pruning: Prunes local receipt files based on receiptPruneAgeDays.
  5. Admin tombstone + cloud orphan cleanup: Time-gated to once per 30 days via lastAdminCleanup in sync prefs. Runs tombstone cleanup and ImageLedgerService.purgeOrphanedCloudFiles().
runMatchingChain(txn: Transaction)
Sequential matching pipeline for incoming transactions (manual or CSV import):
  1. Duplicate check: Calls findDuplicate() with matchDays, matchPercent, matchDollar, matchChars tolerances. If found, shows DuplicateResolutionDialog.
  2. Recurring expense match: Calls findRecurringExpenseMatch(). If matched, shows RecurringExpenseConfirmDialog. On confirm, sets linkedRecurringExpenseId and stores linkedRecurringExpenseAmount (the RE amount at link time).
  3. Amortization match: Calls findAmortizationMatch(). If matched, shows AmortizationConfirmDialog. On confirm, sets linkedAmortizationEntryId.
  4. Budget income match: Calls findBudgetIncomeMatch(). If matched, shows BudgetIncomeConfirmDialog. On confirm, sets isBudgetIncome=true, linkedIncomeSourceId, and linkedIncomeSourceAmount.

On app launch, delegates to PeriodRefreshService.refreshIfNeeded() which:
  1. Loads all data from disk, computes budgetAmount via BudgetCalculator.computeFullBudgetAmount().
  2. Detects missed periods between lastRefreshDate and now.
  3. For each missed period: creates PeriodLedgerEntry (dedup check), accrues SG totalSavedSoFar and RE setAsideSoFar.
  4. Saves to disk, recomputes cash, updates SharedPrefs.
  5. Returns RefreshResult with only changed records (for Firestore push by caller).
When in a sync group, period refresh is deferred until after awaitInitialSync() completes (suspends the coroutine, does not block UI) to allow incoming period ledger entries from other devices to arrive first. Solo users skip the wait. PeriodRefreshService returns null if no periods are missed. The same service is called by BackgroundSyncWorker for background refresh. v2.5: The period refresh loop now uses calculated sleep: computes nextBoundary from BudgetCalculator.currentPeriodStart() plus one period, then sleeps until nextBoundary + 60s with a minimum of 60 seconds. This replaces the previous fixed delay(30_000).
Save Functions Pattern (v2.2+, hint support v2.3+)
All save functions follow the same pattern:
  1. Call the repository's save(context, data) to persist to local JSON.
  2. If SyncWriteHelper is initialized (device in sync group), push records via the corresponding SyncWriteHelper.pushXxx() method. If hint is provided, push exactly those records. If no hint, SyncWriteHelper pushes all records and FirestoreDocSync's diff-based pushRecord() only sends changed enc_* fields.
Transaction Archiving (v2.5)
archiveThreshold (default 10,000; 0 = disabled) controls when old transactions are archived. When the active transaction count exceeds the threshold, applyArchiveCutoff() computes a cutoff date that would archive approximately 25% of the oldest transactions. Archived transactions are moved to a separate archive file and excluded from the main transactions list. The cutoff date is stored in SharedSettings.archiveCutoffDate (synced to all devices). Archived transactions remain viewable but are not editable. Incoming archiveCutoffDate from another device via sync is applied locally if it is later than the current cutoff.
Tombstone Cleanup
v2.5: Moved from startSyncSetup() into runPeriodicMaintenance() (24h-gated). The admin device cleans up old tombstones, time-gated to 30 days via lastAdminCleanup in sync prefs. Uses syncDevices (maintained via RTDB presence) for the oldest device lastSeen timestamp with a 1-day buffer. Deletes Firestore documents where deleted=true and the doc is older than that cutoff. Also runs orphan cloud cleanup via ImageLedgerService.purgeOrphanedCloudFiles(). Local JSON is also purged of tombstones on first launch after group join.

Filtered Listeners (v2.4)

attachCollectionListener() loads the per-collection cursor from SharedPreferences ("sync_cursor" prefs, keys like cursor_transactions_seconds and cursor_transactions_nanos). If a cursor exists, it calls FirestoreDocService.listenToCollectionSince() with whereGreaterThan("updatedAt", cursor). This means only documents modified since the last successful batch delivery are read, dramatically reducing Firestore read costs on app resume. If no cursor exists (fresh install, reinstall, or first sync after upgrade), falls back to unfiltered listenToCollection(). The cursor is saved (via saveCursor()) after onBatchChanged completes, using the latest updatedAt from the batch. Auto-reconnect on listener error uses 5-second delay before reattaching.

Initial Sync Gate (v2.4)

markCollectionDelivered(collection) is called at the top of handleCollectionChanges and handleSharedSettingsChange. When all 8 collections (7 data + sharedSettings) have been delivered, the CompletableDeferred completes and awaitInitialSync() returns. This replaces the old fixed ~200ms delay with a deterministic gate.

Performance Optimizations

  • Filtered listeners (v2.4): Per-collection updatedAt cursors ensure only recently-changed docs are read on reconnect.
  • Enc hash skip: Composite hash of all enc_* fields; persisted to disk (enc_hash_cache.json) so cold starts also skip decryption of unchanged docs.
  • Three-tier threading: Decrypt on Default dispatcher, UI on Main, JSON save on IO.
  • Echo suppression: 5-second window (ECHO_SUPPRESS_MS). Recently-pushed doc keys filtered out of listener callbacks.
  • Diff-based updates: Only changed enc_* fields sent via updateFields(), not full document rewrites.
  • Persistent listeners: Managed by ViewModel lifecycle, not DisposableEffect. No re-delivery on warm resume. Auto-reconnect on fatal error (5s delay).
  • Batched listener callbacks: Save once per collection, not per record.

Conflict Detection

lastEditBy plain field tracks who last edited each doc. localPendingEdits (persisted to SharedPreferences) tracks unpushed offline edits. If listener receives change with lastEditBy != us AND we have pending edit, the event is flagged as conflict. Conflicted transactions are marked isUserCategorized = false.

doWork() Flow — Three-Tier Logic (v2.5)

  1. Tier 1 — Foreground active: Reads MainActivity.isAppActive volatile flag; returns Result.success() immediately if the app is in the foreground (foreground handles its own sync and refresh).
  2. Tier 2 — ViewModel alive (stopped but in memory): Checks MainViewModel.instance?.get(). If the ViewModel is alive and sync is configured, checks if listeners are dead (!ds.isListening) and restarts them via ds.startListeners() on Dispatchers.Main. Returns Result.success() without performing a full sync.
  3. Tier 3 — ViewModel dead (process restarted): Runs full background sync:
  4. Firebase auth: Signs in anonymously (same pattern as DebugDumpWorker).
  5. Firestore sync (if configured): Creates a temporary FirestoreDocSync instance, opens short-lived listeners with ~60-second timeout via CompletableDeferred, accumulates DataChangeEvents, then calls stopListeners().
  6. Process events: Calls SyncMergeProcessor.processBatch(). Saves merged data to disk via repositories. Applies settings to SharedPreferences.
  7. Period refresh: Builds RefreshConfig from SharedPreferences, calls PeriodRefreshService.refreshIfNeeded().
  8. Cash recompute fallback: If sync delivered changes but no period boundary was crossed, calls recomputeCashFromDisk() (private method that loads all data from disk and calls BudgetCalculator.recomputeAvailableCash()) to pick up any sync-delivered transaction changes.
  9. Push to Firestore: Pushes new period ledger entries via createDocIfAbsent(), updated SG via updateFields() with fieldUpdate(sg, setOf("totalSavedSoFar")), updated RE via updateFields() with fieldUpdate(re, setOf("setAsideSoFar", "isAccelerated")).
  10. Push sync side effects: Deletes remapped categories from Firestore, pushes back conflicted transactions.
  11. Update widget + device metadata: Calls BudgetWidgetProvider.updateAllWidgets() and updates device metadata via FirestoreService.updateDeviceMetadata().

Private Methods

recomputeCashFromDisk()

14. Error Handling

14.1 CSV Import Errors

Line-level parse errors with line number and message. Header validation for BudgeTrak CSV format. Empty file detection. Partial results preserved.

14.2 Encryption/Decryption Errors

Minimum data size validation. Poly1305 authentication tag verification. Invalid password/key throws authentication failure. Per-field encryption errors in sync are logged and the record is skipped (hint for wrong encryption key on AEADBadTag).

14.3 Preference Loading Errors

Try-catch with defaults for enum parsing (BudgetPeriod). Multi-type cascade via getDoubleCompat() for numeric values.

14.4 Repository Loading Errors

File existence check before read. Empty/blank content returns emptyList(). Missing fields loaded with backward-compatible defaults. Corrupt records skipped with warning log.

14.5 ID Collision Handling

All ID generators use do-while retry loop. 16-bit ID space (0..65535) with collision avoidance.

14.6 Budget Calculation Safety

Division by zero guarded (periods count > 0). Day-of-month clamped to actual month length. Budget calculations rounded to 2 decimal places. NaN/Infinity guarded by SafeIO.safeDouble().

14.7 UI Input Validation

Required field validation in all dialogs. Decimal place limiting per currency. Date range validation. Amount sum verification for multi-category.

14.8 Sync Error Handling

Firestore operations wrapped in try-catch with 30-second timeout. Listener auto-reconnect after 5-second delay on error. Enc hash skip prevents redundant decryption. pushRecord() falls back to set() on NOT_FOUND. Period ledger uses create-if-absent to prevent overwrites. Crash logger writes to Download/crash_log.txt. RTDB presence gracefully degrades if Firebase RTDB is not configured (getDatabase() returns null). Filtered listeners fall back to unfiltered on null cursor.

14.9 File I/O Safety

Atomic writes via temp file + rename. Per-file ReentrantLock prevents concurrent writes. Rename fallback to copy on filesystems where renameTo() fails.

14.10 Receipt Sync Safety

Upload-first flow (upload before creating ledger entry). Orphan scan with time guard (30-day admin cleanup interval). Speed-based upload assignment with stale detection. Batch recovery cap (50 receipts). Snapshot lifecycle with error states. 14-day pruning uses local 24h skip gate + group-level lastCleanupDate (idempotent, no CAS).
